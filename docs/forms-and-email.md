# Forms & Email

Both forms on the site submit via JavaScript `fetch()` calls to a **Scaleway Function** (`functions/`), which relays the request to **Resend** for actual email delivery / subscriber storage. Scaleway is a French cloud provider ŌĆö chosen deliberately over Cloudflare/AWS/etc. so the relay layer stays under EU jurisdiction, consistent with the rest of the site's EU-hosted services (StaticHost.eu, self-hosted Litlyx).

- **Contact form** ŌĆö markup lives in `content/contact.md`; the submit handler (`POST` to the `contact` function's URL) is in `static/js/main.js`, handled server-side by `functions/contact.js`. The frontend sends `firstName`/`lastName` as separate fields (not pre-joined), so both templates can show them individually. On a valid submission it sends **two** emails via Resend, both templated to match the site's palette (cream background, dark-rose accent, serif heading): the internal notification to `contact@loconsole.eu` (Resend Template `contact-form-notification`, variables `first_name`/`last_name`/`email`/`phone`/`subject`/`message`, each field clearly divided in its own row), and a templated auto-reply confirmation to the submitter (Resend Template `contact-form-confirmation`, variables `name`/`subject`/`message_preview`). The confirmation send is best-effort ŌĆö a failure there doesn't fail the request, since the notification reaching `contact@loconsole.eu` is what actually matters.
- **Newsletter signup** ŌĆö markup and submit handler (`POST` to the `newsletter` function's URL) are both in `layouts/blog/single.html`, handled server-side by `functions/newsletter.js`. On subscribe, the email is added to the single Resend Audience that currently exists in the account ("General", `RESEND_GENERAL_AUDIENCE_ID` ŌĆö a hardcoded constant in `newsletter.js`, not a secret, just a resource ID) via `POST https://api.resend.com/audiences/{id}/contacts` ŌĆö the audience-scoped endpoint, not the flat `POST /contacts` one. (The flat endpoint accepts an `audienceId` body field but silently ignores it ŌĆö verified directly against the API: it returns `201` and creates a contact, but never actually enrolls it in the named audience. This was true even for a valid audience ID, so the original code's audience-enrollment likely never worked correctly.) Since that endpoint's create call is an idempotent upsert (always `201`, even for an existing contact), `newsletter.js` does a `GET /audiences/{id}/contacts/{email}` first to check whether this is a genuinely new subscriber before deciding to send the welcome email (Resend Template `newsletter-welcome`, best-effort). There used to be a second "Blog" audience referenced in the old Supabase code ŌĆö that audience has since been deleted from the Resend account, so it's not used here.
- Both functions have a basic in-memory per-IP rate limit (3 requests/60s, see `functions/lib/rateLimit.js`) ŌĆö resets on cold start since Scaleway scales to zero, so it's weaker than persistent rate limiting would be, but still blunts rapid-fire abuse within a warm instance.
- CORS is a small hardcoded allowlist in `functions/lib/cors.js`: `https://lorenzo.loconsole.eu` (production) and `http://localhost:1313` (local Hugo dev server).

`functions/` is a separate deployable unit from the Hugo site: StaticHost only ever runs `hugo --minify` (per `statichost.yml`) and never touches it. Deploy with `functions/deploy.sh` (requires the `scw` CLI configured, plus a `RESEND_API_KEY` env var ŌĆö see `functions/.env.example`). It also runs automatically via `.github/workflows/deploy-functions.yml` on pushes to `main` that touch `functions/**`.

The two functions have **no npm dependencies at all** ŌĆö they use Node 22's built-in `fetch` and nothing else, so there's no `node_modules` to audit or ship. `functions/dev-server.js` is a tiny (also dependency-free) local test harness: `node dev-server.js contact 8787` or `npm run dev:contact` / `npm run dev:newsletter`, then `curl` it directly.

The Resend API key is the **only** namespace secret needed on Scaleway (set via `deploy.sh`, which calls `scw function namespace create/update`) ŌĆö it never appears in this repo or in the shipped frontend JS. This is a meaningful difference from the old Supabase setup: previously the Supabase anon key was public-by-design in the client JS (that's how Supabase Edge Functions are meant to be called), while the actual Resend key was hidden inside the Supabase function. Now there's no equivalent public key needed at all ŌĆö the frontend calls the function with no `Authorization` header, and Scaleway is the only place that ever sees the Resend key.

Each function gets its own auto-generated HTTPS URL on first deploy (of the form `https://<name>-<slug>.functions.fnc.<region>.scw.cloud`) ŌĆö there's no custom domain involved, which avoids having to point `loconsole.eu`'s DNS at Scaleway. After running `deploy.sh`, paste the two printed URLs into `static/js/main.js` (contact) and `layouts/blog/single.html` (newsletter). If you deploy to a region other than the CLI default (`fr-par`), also update the `connect-src` wildcard in `static/_headers` (currently `https://*.functions.fnc.fr-par.scw.cloud`) to match.

The Resend Templates referenced by ID (`contact-form-notification`, `contact-form-confirmation`, `newsletter-welcome`) are managed in the Resend dashboard (Templates section), not in this repo ŌĆö editing their content/design there takes effect immediately without any redeploy here. There's also a `contact-form-reply` template in Resend, but that one is a manual draft for composing replies by hand ŌĆö it's not referenced by any code here.

To change the backend logic (e.g. swap Resend for another provider, or adjust validation), edit `functions/contact.js` / `newsletter.js` and re-run `functions/deploy.sh` (or push to `main`, which triggers the same via CI).

If you add any new external endpoint here, also update the CSP `connect-src` directive in `static/_headers` or the request will be blocked in production.
